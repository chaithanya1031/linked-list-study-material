01 · The Problem

Why does this even need to exist?

Arrays are great — until you need to insert something in the middle. Because array elements sit in one contiguous block of memory, adding one item means physically sliding every item after it, one slot to the right.

Array: inserting "8" at index 1

Every value from index 1 onward has to shift right to make room. That's O(n) work just to make space.

Linked List: inserting "8" after node 1

Nobody moves. We just create one new car and re-point two couplings. That's O(1) once you're at the spot.

This is the whole reason linked lists exist: cheap insertion and deletion, paid for with a real cost — no jumping straight to an index. Every data structure is a trade-off, and this is the linked list's.

02 · Build Intuition

The mental model: a train, not a parking lot

🚂

An array is a parking lot with numbered spaces — you can walk straight to space #47. A linked list is a train: each car only knows the car directly behind it. To reach the 47th car, you must walk through cars 1 through 46 first.

Train (Linked List) Real Concept
A train carA node — a small object holding data
Cargo inside the carThe node's value
The coupling hook to the next carThe next pointer
The locomotiveThe head pointer — your only entry point
An open hook at the last carA null pointer — "nothing after me"

Notice what the train can't do: you can't teleport to car 47. You also can't see a car's number from outside — you only find out what's next by standing at the current car and reading its hook. That single limitation explains almost every rule about linked lists.

03 · What Actually Happens

Inside the machine

A node is a tiny object living somewhere in RAM. Its home address is essentially random. What makes it a "list" is only that each node stores the address of the next one. Toggle the view below to see both truths at once.

Cars sit in a tidy row, connected by couplings.

Node = 2 fields

Every node stores exactly two things: its value and the address of the next node. That's it — no size field, no index.

head is external

The list itself is really just one pointer variable — head — living in the calling code. Lose it, and every node becomes unreachable garbage.

Scattered, not contiguous

Unlike an array, nodes can live anywhere the memory allocator decides. That's exactly why insertion is cheap and random access is not.

06 · Common Mistakes

Watch it break — on purpose

The lost node

If you overwrite head before saving its old value, every existing node becomes unreachable — still in memory, but nobody can walk to it.

Off-by-one insert

"Insert at index 2" means after the 2nd node. Stop one step too early or late and the value lands in the wrong slot.
